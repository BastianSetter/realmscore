## Punkt 1: Kartensortierung (Farbe, Name) statt (Farbe, Punkte)

### Problem
Karten erscheinen aktuell effektiv nach `(Suit, baseStrength)` sortiert.

> 🔧 **Code-Befund:** Es gibt **keine** explizite `(Suit, baseStrength)`-Sortierung im Code. Die Reihenfolge stammt schlicht aus der **Asset-Datei-Reihenfolge** in `app/src/main/assets/cards/base_game.json` (dort sind die Karten nach Suit gruppiert und innerhalb aufsteigend nach `baseStrength` gelistet). `CardLookup.loadFromAssets()` (`data/cards/CardLookup.kt`) übernimmt diese Reihenfolge 1:1, und der `CardPicker` (`ui/components/CardPicker.kt`) **filtert nur**, sortiert aber nie. Eine Suche nach `sortedBy`/`sortWith`/`compareBy` auf `CardDefinition`-Listen findet daher **nichts Relevantes** – es muss eine Sortierung **neu hinzugefügt** werden, nicht eine bestehende geändert.

### Fix
Eine Sortierung auf `(Suit, nameDe)` **neu einführen**:

```kotlin
cards.sortedWith(compareBy({ it.suit.ordinal }, { it.nameDe }))
```

🔍 **CLAUDE CODE:** Zentral sortieren, damit alle Anzeigeflächen profitieren. Zwei sinnvolle Orte:
- **`CardLookup.getAll()` bzw. `loadFromAssets()`** – sortiert die Liste einmal an der Quelle (deckt `CardPicker`, Sandbox, Necromancer-Kandidaten etc. ab). **Achtung:** `getByKey`/`search`/`filterBySuits`/`getNecromancerEligibleCards` greifen alle auf dieselbe `cards`-Liste zu – einmal an der Quelle sortieren reicht.
- **oder im `CardPicker`** in der `filtered`-Berechnung (`asSequence()....sortedWith(...)`) – betrifft nur den Picker.

Der `CardPicker` ist die einzige Karten-Auswahl-Oberfläche (auch Sandbox nutzt ihn), daher genügt eine der beiden Stellen. `nameDe` als Vergleichsschlüssel ggf. `lowercase()` für saubere alphabetische Reihenfolge.

### Akzeptanzkriterien
- [ ] Karten im CardPicker sind innerhalb jedes Suits alphabetisch nach deutschem Namen sortiert
- [ ] Gilt für alle Stellen die Karten-Listen anzeigen (Sandbox, PlayerHandEntry, CardPicker)

## Punkt 6: Layout – Karten gleichmäßig in Reihen (Solitaire-Style), variable Kartenzahl

### Problem
Karten-Slots werden in einer einzigen horizontalen Reihe dargestellt → bei 7 Karten passen nur ~4,5 auf den Bildschirm, horizontales Scrollen nötig. Bei größeren Mengen (z.B. Mittelfeld mit 10–12 Karten) ist es noch schlimmer.

### Soll
- Karten brechen in **mehrere Reihen** um, gestapelt untereinander wie bei Solitaire
- Es gibt eine ideale Karten Breite, dann wird berechnet, wie viele Karten dieser Breite auf den Screen passen (mit Kommazahl), dieser Wert wird dann gerundet, diese Menge an karten wird dann In einer Reihe **auf den Bildschrim gestreckt** angezeigt, der Rest bricht in die nächste Reihe um
- Funktioniert für **beliebige Kartenzahlen** (7 Handkarten, 10/12 Mittelfeld-Karten, etc.) – nicht auf 7 hartkodiert

```
Beispiel 7 Karten (4 pro Reihe):     Beispiel 12 Karten:
[K][K][K][K]                          [K][K][K][K]
[K][K][K]                             [K][K][K][K]
                                      [K][K][K][K]
```

> 🔧 **Code-Befund (gute Nachricht – das gemeinsame Composable existiert schon):**
>
> - Karten-Slots werden zentral von **`HandSlotsRow`** (`ui/sandbox/components/HandSlotsRow.kt`) gerendert. Es ist **bereits geteilt**: `SandboxScreen` (`:115`) und `PlayerHandCaptureContent` nutzen es; letzteres deckt sowohl die Spieler-Hände als auch das **Mittelfeld** (10/12 Karten via `requiredSlotCount`) ab. Keine Duplizierung.
> - Aktuell ist es ein einzelnes `Row` mit `.horizontalScroll(...)` → genau das gemeldete Problem.
> - Die einzelne Karte `CardSlotView` (`ui/sandbox/components/CardSlot.kt:45`) hat aktuell eine **feste Größe** `size(width = 80.dp, height = 120.dp)`. Für den neuen Soll-Zustand (Spalten berechnen → gestreckt) muss diese feste **Breite** aufgegeben werden: `CardSlotView` soll eine von außen übergebene Breite akzeptieren und die **Höhe per Seitenverhältnis** ableiten (80:120 = 2:3), sonst werden die gestreckten Karten verzerrt. Höhe also über `aspectRatio(80f/120f)` statt fester `height`.

🔍 **CLAUDE CODE:** Zwei Dateien betroffen:
1. **`HandSlotsRow`** – verfügbare Breite messen (`BoxWithConstraints`), daraus Spaltenzahl berechnen (ideale Breite → gerundet), Karten in Reihen dieser Spaltenzahl legen und auf die Breite **strecken**.
2. **`CardSlotView`** (`ui/sandbox/components/CardSlot.kt`) – feste `size(width=80.dp, height=120.dp)` auf „Breite kommt von außen + `aspectRatio`" umstellen, damit gestreckte Karten ihr Seitenverhältnis behalten.

Da `HandSlotsRow` bereits geteilt ist, profitieren Sandbox, Spieler-Hände und Mittelfeld automatisch.

### Umsetzung

**Spaltenzahl aus idealer Breite berechnen, dann auf Bildschirmbreite strecken** (`BoxWithConstraints` + berechnete Kartenbreite):

```kotlin
// HandSlotsRow.kt
import androidx.compose.foundation.layout.BoxWithConstraints
import androidx.compose.foundation.layout.ExperimentalLayoutApi
import androidx.compose.foundation.layout.FlowRow
import androidx.compose.foundation.layout.fillMaxWidth
import androidx.compose.foundation.layout.width
import kotlin.math.roundToInt

private val IdealCardWidth = 80.dp   // ideale Kartenbreite – Basis für die Spaltenzahl
private val CardSpacing = 8.dp

@OptIn(ExperimentalLayoutApi::class)
@Composable
fun HandSlotsRow(
    slots: List<CardSlot>,
    onSlotTap: (Int) -> Unit,
    modifier: Modifier = Modifier,
) {
    BoxWithConstraints(modifier = modifier.fillMaxWidth()) {
        // 1) Wie viele Karten der Ideal-Breite passen (Kommazahl) → gerundet, min. 1
        val columns = (maxWidth / IdealCardWidth).roundToInt().coerceAtLeast(1)
        // 2) Tatsächliche Kartenbreite, sodass `columns` Karten die volle Breite füllen (gestreckt)
        val cardWidth = (maxWidth - CardSpacing * (columns - 1)) / columns

        FlowRow(
            horizontalArrangement = Arrangement.spacedBy(CardSpacing),
            verticalArrangement = Arrangement.spacedBy(CardSpacing),
            maxItemsInEachRow = columns,   // erzwingt den Umbruch nach `columns` Karten
        ) {
            slots.forEachIndexed { index, slot ->
                CardSlotView(
                    slot = slot,
                    onClick = { onSlotTap(index) },
                    modifier = Modifier.width(cardWidth),   // gestreckte, für alle gleiche Breite
                )
            }
        }
    }
}
```

> **Wichtig:**
> - `maxItemsInEachRow = columns` + feste `cardWidth` (statt `weight`) → **alle** Karten exakt gleich breit, volle Reihen füllen die Bildschirmbreite. Die letzte (unvollständige) Reihe ist linksbündig und ihre Karten sind genauso breit wie in den vollen Reihen (sie füllt die Breite **nicht** aus – das ist gewollt).
> - `(maxWidth / IdealCardWidth)` ist in Compose `Dp / Dp = Float` → `.roundToInt()` setzt die „auf-/abgerundete" Kartenzahl pro Reihe um (genau wie im Soll beschrieben).
> - `CardSlotView` muss die `Modifier.width(...)` respektieren und seine Höhe per `aspectRatio` ableiten (siehe Code-Befund), sonst überschreibt die bisherige feste `size(...)` die Breite.
> - `FlowRow` benötigt `@OptIn(ExperimentalLayoutApi::class)`. Das bisherige `horizontalScroll`/`rememberScrollState` entfällt.

### Gilt für (automatisch über das geteilte `HandSlotsRow`)
- Hand-Slots in `SandboxScreen`
- Hand-Slots im Spiel-Flow (`PlayerHandCaptureContent` → `RoundCaptureScreen`)
- Mittelfeld-Darstellung (läuft über dieselbe `PlayerHandCaptureContent`)

### Akzeptanzkriterien
- [ ] Karten brechen in mehrere Reihen um statt horizontal zu scrollen
- [ ] Spaltenzahl ergibt sich aus (Bildschirmbreite / idealer Kartenbreite), gerundet
- [ ] Volle Reihen sind auf die Bildschirmbreite gestreckt; alle Karten gleich breit
- [ ] Letzte Reihe ist linksbündig, Karten gleich breit wie in vollen Reihen
- [ ] Karten-Seitenverhältnis bleibt erhalten (keine Verzerrung durch das Strecken)
- [ ] Funktioniert mit variabler Kartenzahl (7, 10, 12, …) ohne Hardcoding
- [ ] Ein wiederverwendbares Composable (`HandSlotsRow`) für alle Stellen

## Punkt 7: Worttrennung – automatische Silbentrennung (API 29+)

### Problem
An verschiedenen Stellen brechen lange deutsche Wörter unschön/falsch um (z.B. "Einstellunge/n", lange Kartennamen oder Begriffe in der Aufschlüsselung).

### Entscheidung
**Automatische Silbentrennung** verwenden. minSdk darf dafür auf **29** angehoben werden.

### Umsetzung

> 🔧 **Code-Befund:** `minSdk = 26` bestätigt (`app/build.gradle.kts:18`). **Wichtig zu `Type.kt`:** Die `Typography`-Definition (`ui/theme/Type.kt`) überschreibt aktuell **nur** `bodyLarge`; alle anderen Styles (`titleLarge`, `titleMedium`, `bodyMedium`, `labelLarge`, …) sind Material-Defaults. Ein `Hyphens.Auto` nur in `bodyLarge` greift daher **nicht app-weit** – z.B. das Bottom-Nav-Label und viele Titel/Chips nutzen andere Styles. Es müssen also entweder die tatsächlich verwendeten Styles explizit ergänzt werden, oder ein bewusster Satz an Styles.

minSdk in `app/build.gradle.kts` auf 29 anheben:
```kotlin
android {
    defaultConfig {
        minSdk = 29   // war 26 - für automatische Hyphenation
    }
}
```

Automatische Trennung über die Typografie aktivieren. In Compose via `TextStyle` mit `Hyphens.Auto`:

```kotlin
// ui/theme/Type.kt – an JEDEM verwendeten Style, nicht nur bodyLarge
val Typography = Typography(
    bodyLarge = TextStyle(
        // ... bestehende Werte ...
        hyphens = Hyphens.Auto,
        lineBreak = LineBreak.Paragraph
    ),
    // titleLarge, titleMedium, bodyMedium, labelLarge, … analog ergänzen
)
```

🔍 **CLAUDE CODE:**
1. Setze `hyphens = Hyphens.Auto` (und sinnvoll `lineBreak`) an den **tatsächlich genutzten** Text-Styles. Da `Type.kt` bisher nur `bodyLarge` definiert, müssen die übrigen Styles ergänzt werden (sonst greift es nicht app-weit). Welche Styles vorkommen, zeigt eine Suche nach `MaterialTheme.typography.` quer durch `ui/`.
2. Locale: Die App setzt das Locale aktuell nicht explizit. Prüfen ob für korrekte deutsche Trennregeln `LocaleList`/`lang` nötig ist (bei reiner Deutsch-App i.d.R. System-Locale, ggf. testen).
3. Prüfe ob durch minSdk 29 irgendwo API-Level-abhängiger Code vereinfacht werden kann (optional). **Hinweis:** Der einzige `Build.VERSION.SDK_INT`-Guard im Code (`ui/theme/Theme.kt:78`, Dynamic Colors) prüft auf API 31 (`S`) und profitiert daher **nicht** von 26→29 – hier ist nichts zu vereinfachen.

> **F-Droid-Hinweis:** minSdk-Änderung ist unkritisch für F-Droid, erfordert aber – falls bereits veröffentlicht – einen neuen versionCode. Da wir noch in der Entwicklung sind: einfach mitziehen.

### Akzeptanzkriterien
- [ ] minSdk auf 29 angehoben, App baut + läuft
- [ ] Automatische Silbentrennung greift app-weit an den zentralen Text-Styles
- [ ] Deutsches Locale sorgt für korrekte Trennregeln
- [ ] Lange Begriffe brechen an sinnvollen Silbengrenzen um (kein hartes Abschneiden mitten im Wort)
