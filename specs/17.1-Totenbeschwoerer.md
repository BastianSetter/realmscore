# Phase 17.1 – Totenbeschwörer als Joker (entkoppelt vom Mittelfeld-Scan)

## Anweisung an Claude Code

Lies diese Datei vollständig. Lies anschließend `00-vision.md`. Setze danach diese Phase vollständig um.

Voraussetzung: Phase 17 (Profilverwaltung) abgeschlossen. Der MVP inkl. Scoring-Engine (Phase 05) und PlayerHandEntry (Phase 08) ist vorhanden.

> **Einordnung:** Diese Phase liegt zwischen 17 und 18. Sie kommt **vor** dem manuellen Mittelfeld-Scan (Phase 20). Deshalb implementiert sie den Totenbeschwörer zunächst mit **voller Kartenauswahl** (funktioniert immer, auch ohne Mittelfeld). Phase 20 ergänzt später den gefilterten Fall (nur erfasste Mittelfeld-Karten).

---

## Kontext (kurz)

Der **Totenbeschwörer** (necromancer) zieht eine Karte aus dem Mittelfeld in die eigene Wertung. Bisher war diese Funktion an den Mittelfeld-Scan gekoppelt – aber der Scan ist optional. Diese Phase entkoppelt den Totenbeschwörer: Er wird wie ein **Joker** behandelt und bekommt ein eigenes Auswahlfeld unter der Hand, genau wie Doppelgänger/Spiegelung ihr Ziel wählen.

So funktioniert der Totenbeschwörer **immer**, unabhängig davon, ob das Mittelfeld erfasst wurde.

---

## Regeldetails (verbindlich)

- **Einschränkung:** Die geholte Karte darf **kein Magier** (Suit `WIZARD`) sein. (Original-Regel: Totenbeschwörer holt eine Nicht-Magier-Karte aus dem Mittelfeld.)
- **Wertung:** Die geholte Karte wird **zusätzlich** gewertet – die Hand hat dann effektiv **8 Karten** in der Wertung (die 7 Handkarten inkl. Totenbeschwörer + die geholte Karte). Die geholte Karte interagiert voll mit der restlichen Hand (Boni, Strafen, Blanking etc.).
- **Auswahl-Quelle:**
  - **Mittelfeld NICHT gescannt** (Default in dieser Phase): volle CardPicker-UI (alle Nicht-Magier-Karten, abzüglich der bereits in der Hand liegenden)
  - **Mittelfeld gescannt** (kommt mit Phase 20): nur die erfassten Mittelfeld-Karten (gefiltert auf Nicht-Magier), als kurze Liste
- **Optimal-Button:** Berücksichtigt die Totenbeschwörer-Wahl **nur, wenn das Mittelfeld gescannt wurde** (dann ist die Auswahl klein genug für sinnvolles Brute-Force). Ohne Scan bleibt die Totenbeschwörer-Karte bei "Optimal" unangetastet (manuelle Wahl bleibt bestehen).

## Scope

### Drin
- Totenbeschwörer wird als joker-artige Karte behandelt (eigenes Auswahlfeld)
- Auswahlfeld erscheint im PlayerHandEntry **und** in der Sandbox, sobald ein Totenbeschwörer in der Hand liegt
- Volle CardPicker-Auswahl, gefiltert auf Nicht-Magier + nicht bereits in Hand
- ScoringEngine wertet die geholte Karte als 8. Karte
- OptimalSolver: Totenbeschwörer nur bei gescanntem Mittelfeld einbezogen (Hook vorbereiten; aktiver Fall kommt mit Phase 20)
- Persistierung der Totenbeschwörer-Wahl (analog zu Joker-Assignments)

### Explizit NICHT drin
- Kein Mittelfeld-Scan-UI (Phase 20)
- Keine gefilterte "nur Mittelfeld-Karten"-Auswahl (Phase 20 ergänzt das)
- Keine Änderung an anderen Jokern

---

## Was am Ende funktionieren muss

**Klick-Pfad (Sandbox + PlayerHandEntry):**

1. Hand mit Totenbeschwörer befüllen (z.B. als eine der 7 Karten)
2. Unter der Hand erscheint – wie bei anderen Jokern – ein Feld **"Totenbeschwörer holt: [Karte wählen]"**
3. Tap → CardPicker öffnet sich, zeigt alle Nicht-Magier-Karten (abzüglich der Karten in der Hand)
4. Karte wählen → erscheint im Feld
5. Score aktualisiert sich: die geholte Karte wird zusätzlich gewertet (8 Karten in der Wertung)
6. In der Aufschlüsselung / im Ring (Phase 18) taucht die geholte Karte als zusätzliche Karte auf
7. Submit speichert die Totenbeschwörer-Wahl mit
8. **"Optimal"-Button** lässt die Totenbeschwörer-Wahl unangetastet (da Mittelfeld nicht gescannt) – die anderen Joker werden wie gewohnt optimiert

## Architektur-Vorgaben

### Domain: Totenbeschwörer als Erweiterung der Joker-Logik

Der Totenbeschwörer ist bereits über `specialKey = "necromancer"` modelliert (Phase 05). Wir behandeln ihn jetzt als **joker-artig** mit einem eigenen Assignment-Typ.

```kotlin
// domain/scoring/ScoringInput.kt – erweitern
data class ScoringInput(
    val hand: List<CardDefinition>,
    val jokerAssignments: Map<String, JokerAssignment>,
    val discardPile: List<CardDefinition> = emptyList(),
    val necromancerPick: String? = null   // NEU: cardKey der geholten Karte, null = keine Wahl
)
```

**Wichtig zur Modellierung:** Behandle den `necromancerPick` in der Engine so, dass die geholte Karte als zusätzliches Element zur effektiven Hand hinzugefügt wird, BEVOR Boni/Strafen berechnet werden. Die geholte Karte:
- zählt mit ihrem Basiswert
- löst ihre eigenen Boni/Strafen aus
- kann von anderen Karten geblankt werden und andere blanken
- interagiert mit der Hand wie eine normale 8. Karte

> **Achtung Edge Case:** Wenn die geholte Karte selbst ein Joker oder Spezial wäre – das ist durch die Nicht-Magier-Einschränkung teilweise entschärft (Spiegelung/Gestaltenwandler sind WILD, nicht WIZARD; Doppelgänger ist WILD). Entscheide defensiv: Die geholte Karte wird mit ihrem **Basis-Effekt** gewertet, löst aber **keine** verschachtelte Joker-Auswahl aus (kein "Joker holt Joker holt Joker"). Falls eine geholte Karte joker-artig ist, wird sie nur mit Basiswert ohne Joker-Effekt gewertet. Dokumentiere diese Entscheidung im Code-Kommentar.

### CardLookup: Filter-Helfer

### ScoringEngine

Im Order-of-Operations (Phase 05) einen Schritt ergänzen:

```
1. Joker auflösen
2. Buch der Wandlungen
3. NEU: Totenbeschwörer-Karte zur effektiven Hand hinzufügen (falls necromancerPick gesetzt)
4. Strafen-Aufheben
5. Blanking ermitteln (geholte Karte einbezogen)
6. Boni anwenden
7. Strafen anwenden
8. Specials
9. Summe
```

Die geholte Karte erscheint im `ScoringResult.perCard` als eigener Eintrag (damit Aufschlüsselung + Ring sie zeigen). Markiere sie z.B. mit einem Flag:

```kotlin
data class CardScoreResult(
    val cardKey: String,
    val contributedScore: Int,
    val isBlanked: Boolean,
    val isNecromancerPick: Boolean = false,   // NEU
    val effects: List<EffectApplication>
)
```

### UI: Totenbeschwörer-Feld

Analog zur `JokerSection` (Phase 05/08) eine `NecromancerSection`, die nur erscheint, wenn ein Totenbeschwörer in der Hand liegt:

```kotlin
@Composable
fun NecromancerSection(
    pickedCardKey: String?,
    onPickCard: () -> Unit,    // öffnet CardPicker
    onClearPick: () -> Unit,
    cardLookup: CardLookup
) {
    Column {
        Text(stringResource(R.string.necromancer_section_title), style = MaterialTheme.typography.titleMedium)
        OutlinedCard(onClick = onPickCard, modifier = Modifier.fillMaxWidth()) {
            Row(Modifier.padding(12.dp), verticalAlignment = Alignment.CenterVertically) {
                if (pickedCardKey != null) {
                    val card = cardLookup.getByKey(pickedCardKey)
                    Text(card?.nameDe ?: "", Modifier.weight(1f))
                    IconButton(onClick = onClearPick) { Icon(Icons.Default.Close, null) }
                } else {
                    Text(stringResource(R.string.necromancer_pick_hint), Modifier.weight(1f))
                    Icon(Icons.Default.Add, null)
                }
            }
        }
    }
}
```

Im PlayerHandEntryScreen und SandboxScreen: Section einblenden, wenn `state.slots` einen Totenbeschwörer enthält. Beim Tap CardPicker öffnen mit:

### Persistierung

Die Totenbeschwörer-Wahl wird wie ein Joker-Assignment gespeichert. In der `HandCard`-Tabelle (Phase 08) ist der Totenbeschwörer eine der 7 Karten. Die geholte Karte speichern wir an der Totenbeschwörer-`HandCard` über das bestehende Feld `jokerTargetCardKey` (semantisch passend: "Ziel" des Totenbeschwörers).

> Falls `jokerTargetCardKey` für Totenbeschwörer semantisch unsauber wirkt, alternativ ein neues nullable Feld `necromancerPickCardKey` in `HandCardEntity` ergänzen (DB-Migration!). **Empfehlung:** `jokerTargetCardKey` wiederverwenden, spart eine Migration. Dokumentiere die Bedeutung im Code.

## Hinweise

- **Regel-Quelle:** Die genaue Totenbeschwörer-Regel (Nicht-Magier, zusätzliche Wertung) aus der offiziellen Anleitung gegenprüfen. Bei Abweichung User fragen.
- **8-Karten-Wertung sauber testen:** Der häufigste Fehler ist, dass die geholte Karte nicht in alle Bonus-/Strafen-Berechnungen der anderen Karten einfließt. Test: eine Handkarte, die "+X pro Bestie" gibt – die geholte Bestie muss diesen Bonus erhöhen.
