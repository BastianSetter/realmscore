# Punkt 1: Suit-Farben an die echten Kartenfarben anpassen

## Ziel
Aktuell nutzen CardPicker und ausgewählte Karten generische Farben. Sie sollen die **tatsächlichen Farben der Spielkarten** verwenden – sowohl im hellen als auch im dunklen Modus.

## Verbindliche Farbreferenz

Aus der offiziellen Suit-Leiste abgelesen. Diese Werte sind die **Basis (für Light Mode)**:

| Suit (intern)   | Anzeigename (DE) | Hex (Light)  |
|-----------------|------------------|--------------|
| `LAND`          | Land             | `#6B4A2B`    |
| `FLOOD`         | Flut             | `#2E4A8A`    |
| `WEATHER`/`FLAME` siehe unten | | |
| Wetter          | Wetter           | `#7FA8C9`    |
| Flamme          | Flamme           | `#C0392B`    |
| Armee           | Armee            | `#3A3A3A`    |
| Zauberer        | Zauberer (Magier)| `#C0398C`    |
| Anführer        | Anführer (Heerführer) | `#7A3FA0` |
| Bestie          | Bestie           | `#4A8A3F`    |
| Waffe           | Waffe            | `#8A8F98`    |
| Artefakt        | Artefakt         | `#D67A2E`    |
| Joker           | Joker            | `#C9A23F`    |

> **Wichtig – Suit-Mapping klären:** Das Spiel hat im Grundspiel 10 Suits/Kategorien. Die obige Leiste zeigt 12 Felder (inkl. Wetter, Flamme, Flut, Artefakt). Beim Umsetzen die internen `Suit`-Enum-Werte (aus Phase 04/05) **1:1** auf diese Anzeigefarben mappen. Falls ein Enum-Wert in der Tabelle fehlt oder ein Name nicht eindeutig zuzuordnen ist: **nicht raten, sondern beim User nachfragen** (Mapping Enum → Anzeigename → Farbe einmal explizit bestätigen lassen).

## Dark-Mode-Ableitung

Die obigen Farben sind für Light Mode. Für Dark Mode **nicht** dieselben Werte nehmen (zu dunkel/zu satt auf dunklem Grund). Regel:
- Sättigung leicht reduzieren, Helligkeit anheben, sodass der Farbton erkennbar bleibt, aber auf dunklem Hintergrund angenehm ist
- Praktischer Ansatz: jede Light-Farbe Richtung Weiß aufhellen (z.B. 25–35 % gegen `#FFFFFF` mischen) und ggf. Sättigung um ~10 % senken
- Ziel: gleicher erkennbarer Farbton, WCAG-tauglicher Kontrast gegen Text

Konkrete Dark-Werte einmal berechnen und als feste Konstanten ablegen (nicht zur Laufzeit mischen, damit es deterministisch bleibt).

## Umsetzung

```kotlin
// ui/theme/SuitColors.kt (neu)

    // Light Mode

    val floodLight = Color(0xFF2E4A8A)

    val flameLight = Color(0xFFC0392B)
    val armyLight = Color(0xFF3A3A3A)

    val leaderLight = Color(0xFF7A3FA0)
    val beastLight = Color(0xFF4A8A3F)
    val weaponLight = Color(0xFF8A8F98)
    val artifactLight = Color(0xFFD67A2E)
    val wildLight = Color(0xFFC9A23F)

    // Dark Mode (aufgehellte Varianten - konkrete Werte beim Umsetzen festlegen)
    // val landDark = Color(0xFF...)  usw.

```

Dann an allen Stellen, die bisher eine Suit-Farbe verwenden, `SuitColors.forSuit(suit, isSystemInDarkTheme())` nutzen:
- CardPicker-Items (Phase 04)
- ausgewählte Karten in den Slots (Sandbox + PlayerHandEntry)
- Ring-Visualisierung Karten-Nodes (Phase 18, `suitColor(card.suit)` ersetzen)

## Akzeptanzkriterien P1
- [ ] Alle Suits nutzen die echten Kartenfarben im Light Mode (Werte aus Tabelle)
- [ ] Dark Mode nutzt aufgehellte, erkennbare Varianten (nicht identisch mit Light)
- [ ] CardPicker, Karten-Slots und Ring-Visualisierung verwenden dieselbe zentrale `SuitColors`-Quelle
- [ ] Kein Hardcoded-Farbwert mehr verstreut im Code – alles über `SuitColors`
- [ ] Suit→Farbe-Mapping mit User bestätigt (bei Unklarheit nachgefragt)

# Punkt 2: Karten-Erfassung als Vollbild-Flow mit Spieler-Dropdown

## Ziel
Der bisherige Flow (RoundEntryScreen mit Spielerliste → Tap auf Spieler → PlayerHandEntry) wird ersetzt durch einen **durchgehenden Vollbild-Erfassungs-Flow**:

- Tap auf "Neue Runde" → direkt im **Karten-Erfassen-Screen** (kein Zwischen-Listen-Screen)
- Vollbild, mit **Dropdown oben**, das den aktuell erfassten Spieler zeigt/wählt
- Nach "Fertig erfasst" → **automatisch zum nächsten Spieler** weiterschalten
- Über das Dropdown kann jederzeit manuell zu einem anderen Spieler gewechselt werden (zum Korrigieren)
- Die **Scan-Reihenfolge der letzten Runde wird gemerkt** und als Default-Reihenfolge verwendet

## Was bestehen bleibt
- Der innere Inhalt des Karten-Erfassens (7 Slots, Joker-Section, Totenbeschwörer-Section, Submit-Logik, Spannungs-Modus ohne Score) bleibt **unverändert** – nur der umgebende Rahmen + Navigation ändern sich.
- Die Persistierung pro Spieler (`HandCardRepository.saveHand`) bleibt gleich.

## Änderung am Flow

**Vorher (Phase 07/08):**
```
GameInProgress → "Neue Runde" → RoundEntryScreen (Spielerliste) → Tap → PlayerHandEntryScreen → Submit → zurück zu RoundEntryScreen
```

**Nachher:**
```
GameInProgress → "Neue Runde" → RoundCaptureScreen (Vollbild, Dropdown) → Submit → auto-next → ... → letzter Spieler Submit → "Punkte enthüllen"
```

Der `RoundEntryScreen` als separater Listen-Screen entfällt im Flow. Seine Rolle (Übersicht, wer schon erfasst ist + Reveal-Button) wandert in den neuen Screen bzw. wird durch den Dropdown-Status + einen finalen Reveal-Button ersetzt.

> **Hinweis:** `RoundEntryScreen` muss nicht zwingend gelöscht werden – aber er wird aus dem Haupt-Flow entfernt. Falls er an anderer Stelle referenziert wird, dort auf den neuen Screen umbiegen.

## Neuer Screen: RoundCaptureScreen

```kotlin
@Composable
fun RoundCaptureScreen(
    roundId: String,
    onAllPlayersCaptured: () -> Unit,   // führt zu Reveal
    onBack: () -> Unit
) {
    val vm: RoundCaptureViewModel = viewModel(...)
    val state by vm.uiState.collectAsState()

    Scaffold(
        topBar = {
            // Spieler-Dropdown statt Titel

                players = state.orderedPlayers,
                currentProfileId = state.currentProfileId,
                captureStatus = state.statusByProfile,   // wer ist fertig
                onSelectPlayer = vm::switchToPlayer
            )
        }
    ) { padding ->
        // Der bisherige PlayerHandEntry-Inhalt (7 Slots, Joker, Totenbeschwörer)
        // für state.currentProfileId, eingebettet statt als eigener Screen
        PlayerHandCaptureContent(
            modifier = Modifier.padding(padding),
            playerState = state.currentPlayerHand,
            onCardChange = vm::...,
            onSubmit = { vm.submitCurrentAndAdvance(onAllPlayersCaptured) }
        )
    }

```

### PlayerDropdown
- Zeigt den aktuellen Spieler prominent (Name + Farbe)
- Aufklappbar: Liste aller Spieler in Scan-Reihenfolge
- Pro Eintrag ein Status-Indikator: erfasst ✓ / offen
- Auswahl wechselt den aktuell bearbeiteten Spieler (zum Korrigieren), ohne den aktuellen Stand zu verlieren

### Auto-Advance-Logik
```kotlin
fun submitCurrentAndAdvance(onAllDone: () -> Unit) {
    viewModelScope.launch {
        // 1. aktuelle Hand speichern (wie bisher saveHand)
        // 2. Scan-Reihenfolge-Index dieses Spielers festhalten (für lastScanOrder)
        // 3. nächsten NOCH NICHT erfassten Spieler in der Reihenfolge wählen
        // 4. wenn keiner mehr offen: onAllDone() (-> Reveal)
        //    sonst: currentProfileId = nächster
    }
}
```

## Scan-Reihenfolge merken (nutzt bestehendes Feld!)

Das Datenmodell hat bereits `GameParticipant.lastScanOrder: Int?` (Phase 06, war als "wird später genutzt" markiert). **Jetzt wird es genutzt:**

- Beim Erfassen eines Spielers: dessen Position in der aktuellen Scan-Reihenfolge in `lastScanOrder` speichern (0 = zuerst erfasst, 1 = als zweites, ...)
- Beim Start einer **neuen** Runde: die Spieler nach `lastScanOrder` aufsteigend sortieren → das ist die Default-Reihenfolge
- Spieler ohne `lastScanOrder` (z.B. brandneu) ans Ende, sortiert nach `seatOrder`

```kotlin
// RoundCaptureViewModel beim Laden:
val orderedPlayers = participants.sortedWith(
    compareBy(nullsLast()) { it.lastScanOrder } then compareBy { it.seatOrder }
)
```

Beim Abschluss jeder Runde `lastScanOrder` für alle Teilnehmer aktualisieren (Repository-Methode ergänzen):
```kotlin
interface GameRepository {
    suspend fun updateScanOrder(gameId: String, profileIdToOrder: Map<String, Int>)
}
```

## Akzeptanzkriterien P2
- [ ] "Neue Runde" führt direkt in den Vollbild-RoundCaptureScreen (kein Listen-Zwischenscreen)
- [ ] Spieler-Dropdown oben zeigt aktuellen Spieler + Status aller Spieler
- [ ] "Fertig erfasst" speichert + schaltet automatisch zum nächsten offenen Spieler
- [ ] Über Dropdown kann manuell zu beliebigem Spieler gewechselt werden (auch bereits erfasste, zum Korrigieren)
- [ ] Nach dem letzten Spieler erscheint der Weg zu "Punkte enthüllen" (Reveal)
- [ ] Scan-Reihenfolge wird in `lastScanOrder` gespeichert
- [ ] Neue Runde nutzt die `lastScanOrder` der Vorrunde als Default-Reihenfolge
- [ ] Spannungs-Modus bleibt: kein Score während der Erfassung sichtbar
- [ ] Der innere Erfassungs-Inhalt (Slots/Joker/Totenbeschwörer) funktioniert wie vorher
- [ ] Bestehende Persistierung (saveHand) unverändert
- [ ] Wechsel zwischen Spielern verliert keine eingegebenen Karten des aktuellen Spielers

## Hinweise P2
- **Größter Eingriff dieser Sammlung** – isoliert testen.
- Den bestehenden `PlayerHandEntry`-Inhalt möglichst als wiederverwendbares Composable extrahieren (`PlayerHandCaptureContent`), statt zu duplizieren.
- Edge Case: Wenn man über das Dropdown zu einem schon erfassten Spieler springt und ihn ändert → erneutes Submit aktualisiert seinen Eintrag, Auto-Advance springt dann zum nächsten **offenen** Spieler (nicht stur +1).
- Edge Case: alle Spieler bereits erfasst, User springt nochmal rein zum Korrigieren → nach Submit direkt wieder Reveal-Option anbieten.

# Punkt 3: Bessere Autovervollständigung bei der Spielerauswahl (NewGame)

## Ziel
Bei der Spielerauswahl im NewGameScreen (Phase 06):
1. Vorschläge erscheinen **schon ohne ersten Buchstaben** (sobald das Feld fokussiert wird / leer ist)
2. Die **Reihenfolge** der Vorschläge ergibt sich aus einer Relevanz-Metrik basierend auf **Anzahl** und **Aktualität** der gemeinsamen Spiele

## Änderung an der Autocomplete-Logik

**Vorher (Phase 06):** `searchByNamePrefix(query)` – liefert erst bei Eingabe etwas, sortiert alphabetisch.

**Nachher:**
- Bei leerem Query: alle (nicht-archivierten, nicht bereits gewählten) Profile vorschlagen, sortiert nach Relevanz-Score
- Bei Eingabe: gefiltert auf Präfix-Match, aber innerhalb der Treffer ebenfalls nach Relevanz sortiert

## Relevanz-Metrik

Pro Profil ein Score aus zwei Faktoren:
- **Häufigkeit:** Anzahl der Spiele, in denen dieses Profil (mit dem Owner) gespielt hat
- **Aktualität:** wie kürzlich das letzte gemeinsame Spiel war

Vorschlag für die Formel (beim Umsetzen ggf. justieren):
```kotlin
// recencyWeight: jüngere Spiele zählen mehr (exponentieller Abfall)
// score = Summe über alle gemeinsamen Spiele von: exp(-ageInDays / HALF_LIFE)
// HALF_LIFE z.B. 30 Tage
// Profile mit mehr UND aktuelleren Spielen ranken oben.
fun relevanceScore(games: List<GameWithDate>): Double {
    val halfLifeDays = 30.0
    return games.sumOf { game ->
        val ageDays = (now - game.startedAt) / 86_400_000.0
        Math.exp(-ageDays / halfLifeDays)
    }
}
```

So schlägt ein Profil, mit dem man oft und kürzlich gespielt hat, weiter oben vor als eines, das man einmal vor einem Jahr eingetragen hat.

## Repository-Erweiterung
```kotlin
interface ProfileRepository {
    // NEU: Profile mit Relevanz-Score, optional gefiltert per Präfix
    suspend fun suggestProfiles(
        prefix: String,           // "" = alle
        excludeProfileIds: Set<String>,   // bereits gewählte + Owner
        ownerId: String
    ): List<Profile>   // bereits nach Relevanz sortiert
}
```
Die Relevanz-Berechnung greift auf die Spiel-/Teilnehmer-Tabellen zu (gemeinsame Spiele mit dem Owner).

## UI-Anpassung (NewGameScreen)
- Das Vorschlags-Dropdown öffnet sich auch bei leerem Feld (on focus)
- Debounce wie bisher (150 ms) bei Eingabe
- Liste zeigt die nach Relevanz sortierten Profile

## Akzeptanzkriterien P3
- [ ] Bei Fokus auf das leere Spieler-Feld erscheinen sofort Vorschläge (ohne Tippen)
- [ ] Vorschläge sind nach Relevanz sortiert (Häufigkeit + Aktualität), nicht alphabetisch
- [ ] Profil mit vielen/kürzlichen gemeinsamen Spielen steht oben
- [ ] Bei Eingabe wird auf Präfix gefiltert, Reihenfolge bleibt relevanz-basiert
- [ ] Bereits gewählte Spieler + Owner werden nicht vorgeschlagen
- [ ] Archivierte Profile (Phase 17) werden nicht vorgeschlagen
- [ ] Performance: Vorschläge erscheinen ohne spürbare Verzögerung (ggf. Relevanz im Hintergrund berechnen/cachen)

## Hinweise P3
- Die Metrik nicht überoptimieren – die exp-Abfall-Formel ist ein guter Startpunkt, `halfLifeDays` ist der Stellhebel.
- Falls noch keine Spielhistorie existiert (frische App): Fallback auf alphabetische Sortierung, kein Fehler.
